## Background

いまなぜこれを作るのか。どんな状況・文脈があるか。
既存プロダクトのリニューアルなら、前バージョンの経緯もここ。

## Who

ロール名の列挙ではなく、場面の描写。
「どんな状態の人が、どんな場面で使うか」を書く。
同じ人でもシーンが違えば分けて書く。

## Problem

何が困っているか。根っこの問題を書く。
「通知がない」は Problem ではなく Solution の不在。「期限が過ぎてから気づく」が Problem。
Who ごとに問題が違うなら分けて書く。

## Solution

機能一覧ではなく、どういう方向で解くか。やらない判断もここ。
既存プロダクトのリニューアルなら前バージョンとの違い。
主要なユーザフローがあれば 1-2 個書く（Ticket の Acceptance Criteria を導く素材になる）。

## Appetite

このプロダクトにどこまで投資する価値があるか（時間・複雑さの上限感）。1〜3 行でよい。
coding agent はここを YAGNI 判断の較正に使う（例:「小さく保つ。大工事になる解決策は提案止まりにする」）。
書けない場合はセクションごと省略してよい。

## Constraints

すべての判断を規定する前提条件。
技術スタック、対応環境、認証方式、開発体制、外部依存。
coding agent はここを見て「使っていい技術」「変えられない前提」を知る。
暗黙知を明示する場所。「当然でしょ」と思うことも書く。
実装時に落とし穴になりうる既知の地雷（rabbit holes）もここに書く。

## Architectural Invariants

実装がどう変わっても壊してはならない構造的な約束。`AI-1` 形式の番号付き 1 項目 1 行で書く（検索・照合可能にするため）。
例: データ隔離の方式、認証情報の受け渡し方、秘密情報を置いてよい場所。
Ticket は実装前にこの一覧との整合を宣言し、違反が必要ならコードではなく本セクションの改訂を先に合意する。
変更は破壊的影響が大きいため、ユーザ承認が必須。

## Done

うまくいったと言える状態。自分が判断できる言葉で。
計測しないメトリクスは書かない。
フォーマットは自由。散文でも箇条書きでもよい。
達成した項目は消さず、日付または commit / ticket 名を添えて「達成済み」と追記する（「現在」「最近」で済ませない）。

## Non-goals

やりたくなるが、意図的にやらないこと。
「議論した上でやらない」と決めたことを書く。
議論にすらならないことは書かない。

## Open Questions

答えが実質出ているものは書かない。
coding agent はここに該当する判断を勝手にしない。
